Описание приложения «Info Industries Group Auto Reporter» для заявки на доступ к API Яндекс.Директа

Назначение
Приложение формирует еженедельный сводный отчёт по эффективности рекламных кампаний
клиентов агентства и отправляет его во внутренние рабочие чаты. Приложение работает
только в режиме чтения статистики и не создаёт, не изменяет и не удаляет объекты в Директе.

1. Используемые методы API
— Сервис Reports (https://api.direct.yandex.com/json/v5/reports), API версии 5, формат JSON,
  тело отчёта — TSV.
  Тип отчёта: ACCOUNT_PERFORMANCE_REPORT.
  Запрашиваемые поля (FieldNames): Impressions, Clicks, Cost, Conversions.
  DateRangeType: CUSTOM_DATE (период — предыдущая календарная неделя, понедельник–воскресенье).
Других сервисов и методов приложение не вызывает (Campaigns, AdGroups, Ads, Keywords,
Dictionaries и прочие не используются). Изменяющие методы (add/update/delete) не используются.

2. Схема и последовательность вызова методов
1) Запуск по расписанию — один раз в неделю (понедельник, 09:00).
2) Приложение читает локальный список клиентов (логины) из конфигурационного файла.
3) Для каждого клиента строго последовательно:
   а) один POST-запрос к сервису Reports с заголовком Client-Login = логин клиента;
   б) если сервер ставит отчёт в очередь (HTTP 201/202) — приложение ждёт время, указанное
      в заголовке retryIn, и повторяет запрос до получения готового отчёта (HTTP 200);
   в) готовый TSV разбирается, рассчитываются итоговые показатели (расход, показы, клики,
      CTR, CPC, конверсии, CR, CPA);
   г) результат отправляется во внешний сервис уведомлений (Telegram) — к API Директа
      это обращение отношения не имеет.
4) Переход к следующему клиенту.
Параллельные запросы не используются — обращения идут одно за другим.

3. Частота вызова методов
— Сервис Reports: примерно 1 запрос на одного клиента в неделю, плюс редкие повторные
  опросы статуса очереди в соответствии с заголовком retryIn.
— При ~34 клиентах это порядка 34 запросов отчётов в неделю, последовательно, с интервалом
  в несколько секунд между клиентами.
— Никакие методы не вызываются чаще, чем раз в неделю на клиента. Циклов опроса в реальном
  времени и фоновых периодических обращений нет.

4. Обработка ошибок
— Анализируется HTTP-код ответа сервиса Reports.
— HTTP 200 — отчёт готов, тело разбирается.
— HTTP 201/202 — отчёт формируется в очереди; приложение ожидает интервал из заголовка
  retryIn и повторяет запрос (с ограниченным числом попыток), не опрашивая сервер чаще,
  чем указано сервером.
— HTTP 4xx/5xx — приложение читает тело ответа, логирует поля error_code и error_string,
  пропускает текущего клиента и продолжает обработку остальных; сбой по одному клиенту
  не прерывает весь прогон.
— По завершении прогона при наличии ошибок отправляется уведомление администратору
  со списком проблемных клиентов и текстами ошибок. Все события пишутся в локальный лог.

5. Учёт текущих ограничений API Директа
— Используется «лёгкий» агрегированный отчёт ACCOUNT_PERFORMANCE_REPORT (одна итоговая
  строка на клиента) без детализации по дням, кампаниям или фразам — минимальная нагрузка
  и экономный расход баллов.
— Один отчёт на клиента в неделю; объём запросов заведомо в пределах лимитов и баланса баллов.
— Соблюдается механизм отложенных отчётов: при статусах 201/202 приложение ждёт ровно столько,
  сколько указано в retryIn, без частого повторного опроса очереди.
— Запросы выполняются последовательно, без параллелизма; число повторных попыток ограничено.
— Приложение работает только на чтение и не вносит изменений в рекламные кампании.
